# Order-bias validity (standardized local benchmark)

This report checks whether models are overly sensitive to candidate image position
(`shape_first` vs `texture_first`) in the standardized local benchmark.

## Why this gate exists

A model can show good aggregate shape/texture balance while still using a shortcut:
prefer whichever option appears first. This is a validity risk for interpretation.

## Metrics

- `shape_rate_shape_first`: shape-choice rate when shape image is presented first.
- `shape_rate_texture_first`: shape-choice rate when texture image is presented first.
- `order_gap_abs`: absolute difference `|shape_rate_shape_first - shape_rate_texture_first|`.
- `first_option_rate`: rate of choosing the first option, regardless of whether it is shape or texture.

## Labeling rule (default thresholds)

- `valid`: low order gap and no strong first-option bias.
- `borderline`: moderate gap / moderate first-option bias.
- `invalid`: large gap or strong first-option bias.

You can tune thresholds via:

```bash
python scripts/compute_order_bias_validity.py --gap-valid-max 0.15 --gap-borderline-max 0.35
```

## Metric definitions and interpretation

This section is explanatory (not auto-generated) and describes how each metric is
computed from trial rows.

### Main validity metrics (same as `interpret/models_validity.md`)

- `image_tracking_rate`
  - Computed over paired deterministic trials for the same `(model, stim_id, word)`:
    one `shape_first` row and one `texture_first` row.
  - A pair counts as image-tracking success (`1`) when answers flip with order:
    - `shape_first` parsed answer is `1` and `texture_first` parsed answer is `2`, **or**
    - `shape_first` parsed answer is `2` and `texture_first` parsed answer is `1`.
  - Intuition: the model is following the target image identity, not just position.

- `word_sensitivity_rate`
  - Group rows by `(model, stim_id, ordering)` across different words. It operationalizes “would the answer change for same images + same order, if only the word changes?”
  - Mark group as sensitive (`1`) if both `shape` and `texture` choices appear across words.
  - Intuition: changing only the word can change the model decision.
  - One nuance: it’s an aggregate group-level measure, not a single pairwise comparison. It asks whether there exists any change across the tested word set for that fixed stimulus/order context.

- `parse_quality`
  - `parse_quality = 1 - unclear_rate`.
  - `unclear_rate` is the fraction of rows whose normalized choice is `unclear`.
  - Intuition: output is consistently parseable into a valid 2AFC decision.

### Position-bias extension metrics

- `shape_rate_shape_first`
  - Fraction of `shape_first` rows with final `choice == shape`.

- `shape_rate_texture_first`
  - Fraction of `texture_first` rows with final `choice == shape`.

- `order_gap_abs`
  - `|shape_rate_shape_first - shape_rate_texture_first|`.
  - Large values indicate strong dependence on which side/order the options are shown.

- `first_option_rate`
  - In `shape_first` rows: choosing `shape` means choosing option #1.
  - In `texture_first` rows: choosing `texture` means choosing option #1.
  - `first_option_rate` is the fraction of all deterministic rows that choose option #1.
  - Intuition: direct positional shortcut signal (first-option preference).

### Are side/position bias and image tracking the same?

No. They are related but not identical.

- `image_tracking_rate` asks: **does the model keep following the same image identity across order flips?**
- Position-bias metrics ask: **does the model prefer a position (especially first option), regardless of identity?**

Examples:

- A model can have **high position bias** and therefore **low image tracking** (typical shortcut failure).
- A model can have **low position bias** but still **low image tracking** if it is noisy/confused for other reasons.
- A model with **high image tracking** usually has low harmful position bias, but this is empirical, not guaranteed by definition.

So both views are useful: main validity checks task fidelity and parseability; the extension isolates positional shortcut behavior.
